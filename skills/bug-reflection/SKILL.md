# Bug 反思循环 (Bug Reflection)

> **每次发现 bug，都是一次学习机会。用 5 Whys 找根因，用四维分类定改进。**

---

## 触发时机

- 发现 bug 时自动触发
- 测试失败时
- 线上问题排查时
- 用户反馈问题时

---

## 5 Whys 根因分析

对每个 bug，连续问 5 次"为什么"，直到找到根本原因：

```
Bug: 视频列表加载失败
  │
  ▼ Why 1: 为什么加载失败？
  → API 返回 500 错误
  │
  ▼ Why 2: 为什么 API 返回 500？
  → 数据库查询超时
  │
  ▼ Why 3: 为什么数据库查询超时？
  → 查询没有加索引
  │
  ▼ Why 4: 为什么没有加索引？
  → 开发时没考虑性能
  │
  ▼ Why 5: 为什么没考虑性能？
  → 没有性能测试流程
  │
  ▼ 根因：缺少性能测试流程
```

---

## 四维分类

| 维度 | 检查问题 | 改进方向 |
|------|----------|----------|
| **PRD 层面** | 需求写清楚了吗？边界条件定义了吗？ | 完善需求文档 |
| **代码层面** | 实现有逻辑错误吗？ | 修复代码 |
| **测试层面** | 测试覆盖到了吗？Mock 正确吗？ | 补充测试 |
| **流程层面** | 质量节拍的哪一步没执行到位？ | 改进流程 |

---

## 改进措施模板

```markdown
## Bug 反思记录

### Bug 描述
[简要描述 bug]

### 5 Whys 分析
1. Why: [原因1]
2. Why: [原因2]
3. Why: [原因3]
4. Why: [原因4]
5. Why: [根因]

### 四维分类
- PRD: [是否清楚]
- 代码: [是否有错]
- 测试: [是否覆盖]
- 流程: [哪步缺失]

### 改进措施
1. [具体文件变更1]
2. [具体文件变更2]
3. [流程改进]

### 验证
- [ ] 代码已修复
- [ ] 测试已补充
- [ ] 文档已更新
- [ ] 已写入 learnings
```

---

## 集成到日常循环

Step ⑥ 在完成一个子任务后，自动检查当前会话有没有发现新的 bug，有则触发反思循环。
